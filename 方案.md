# Browser Group Env 技术方案

## 1. 总体方案

插件以 Chrome Tab Group 为上下文边界，为不同 Tab Group 注入不同 HTTP 规则。用户仍然使用同一个 Chrome Profile，因此登录态、Cookie、LocalStorage 保持共享；环境差异通过插件规则隔离。

```text
Chrome Profile
  |
  +-- Tab Group A -> Env A -> headers / filters
  +-- Tab Group B -> Env B -> headers / filters
  +-- Tab Group C -> Env A -> headers / filters
```

核心实现：

- 使用 `WXT + React + TypeScript` 构建 Manifest V3 插件工程。
- 使用 `Tailwind CSS` 和无样式组件原语实现现代化 Popup UI。
- 使用 `chrome.tabGroups` 和 `chrome.tabs` 识别 Tab Group 与 Tab。
- 使用 `chrome.declarativeNetRequest.updateSessionRules` 注入 Headers。
- 使用 `chrome.storage.local` 保存 Env、绑定关系和全局配置。
- 使用 popup 页面完成配置编辑。
- 使用 background service worker 监听 Tab / Group / Storage 变化并刷新动态规则。

## 2. 架构

### 2.1 前端技术栈

推荐技术栈：

```text
Runtime / Build:
  WXT
  Vite
  TypeScript

UI:
  React
  Tailwind CSS
  Radix UI primitives
  lucide-react

State / Forms:
  Zustand
  React Hook Form
  Zod

Testing:
  Vitest
  Playwright
```

选型理由：

- `WXT`：面向 Web Extension 的现代工程框架，处理 MV3 多入口、manifest 生成、dev server 和构建产物，比手写 Vite + manifest 更省维护成本。WXT 官方说明其支持 TypeScript，且可通过 Vite 插件接入 React。
- `React + TypeScript`：适合复杂表单、列表、规则编辑和后续状态诊断视图。
- `Tailwind CSS`：插件 UI 尺寸受限，原子化样式能减少样式文件切换成本。
- `Radix UI primitives`：用于 Dialog、Popover、Tabs、Switch、Tooltip 等交互原语，避免自研可访问性细节。
- `Zustand`：Popup 内状态轻量，避免引入 Redux 级别复杂度。
- `React Hook Form + Zod`：Header、Filter、Import JSON 都需要强校验，表单状态应和数据校验绑定。
- `Vitest + Playwright`：规则编译适合单测，插件关键流程适合浏览器级验证。

不建议：

- 不用纯原生 JS + 手写 DOM。这个项目的规则编辑、导入导出、状态诊断会持续增长，原生写法后期维护成本高。
- 不用 Next.js。插件 popup/options/background 不是服务端渲染应用，Next.js 的路由和运行模型不匹配。
- 不在 MVP 引入后端服务。配置先存在 `chrome.storage.local`。

模块边界：

- `entrypoints/background.ts`：只处理 Chrome 事件、状态同步和 DNR 编译调用。
- `entrypoints/popup/*`：只处理 UI 和用户操作。
- `src/extension/*`：封装 Chrome API，避免 UI 直接调用 `chrome.*`。
- `src/model/*`：纯函数和类型，便于单测。
- `src/components/*`：无业务副作用组件。

## 3. 权限

建议 `manifest.json` 权限：

```json
{
  "manifest_version": 3,
  "permissions": [
    "tabs",
    "tabGroups",
    "storage",
    "declarativeNetRequest"
  ],
  "host_permissions": [
    "<all_urls>"
  ],
  "background": {
    "service_worker": "background.js"
  },
  "action": {
    "default_popup": "popup.html"
  }
}
```

说明：

- `tabs`：读取当前 Tab、URL、Window 信息。
- `tabGroups`：读取和聚焦 Tab Group。
- `storage`：保存 Env 配置。
- `declarativeNetRequest`：按规则修改请求 Header。
- `host_permissions`：第一版可用 `<all_urls>`，但实际注入必须受 Env filters 限制。

## 4. 数据模型

### 4.1 GlobalState

```ts
type GlobalState = {
  enabled: boolean;
  autoSwitch: boolean;
  selectedEnvId?: string;
  envs: Record<string, Env>;
  groupBindings: Record<string, GroupBinding>;
  ruleMeta: RuleMeta;
};
```

### 4.2 Env

```ts
type Env = {
  id: string;
  name: string;
  enabled: boolean;
  linkedGroupKeys: string[];
  filters: EnvFilters;
  rules: EnvRules;
  createdAt: number;
  updatedAt: number;
};
```

### 4.3 GroupBinding

Chrome 的 `tabGroupId` 可能不适合作为长期持久化主键。建议使用 `groupKey` 抽象一层。

```ts
type GroupBinding = {
  groupKey: string;
  envId: string;
  chromeGroupId?: number;
  windowId?: number;
  title?: string;
  color?: string;
  lastSeenTabUrls: string[];
  updatedAt: number;
};
```

`groupKey` 可以用以下信息生成：

```text
chromeGroupId 存在时：chrome:<windowId>:<chromeGroupId>
导入或恢复时：title:<normalized-title>:<hash-of-first-url>
```

### 4.4 EnvFilters

```ts
type EnvFilters = {
  domains: string[];
  paths: string[];
  excludedDomains: string[];
};
```

匹配语义：

- `domains` 至少需要一项，否则 Env 不可启用。
- `*.example.com` 匹配子域名，不匹配 `example.com` 本身，除非用户单独配置。
- `paths` 为空表示全部路径。
- `excludedDomains` 优先级最高。

### 4.5 EnvRules

```ts
type EnvRules = {
  headers: HeaderRule[];
  queries: QueryRule[];
  cookies: CookieRule[];
};

type HeaderRule = {
  id: string;
  enabled: boolean;
  name: string;
  value: string;
  description?: string;
};

type QueryRule = {
  id: string;
  enabled: boolean;
  key: string;
  value: string;
  policy: "append" | "replace" | "keep";
};

type CookieRule = {
  id: string;
  enabled: boolean;
  name: string;
  value: string;
  domain: string;
  path: string;
  secure: boolean;
  sameSite: "lax" | "strict" | "none";
  expirationDate?: number;
};
```

MVP 只实现 `HeaderRule` 的实际注入。

## 5. Header 注入方案

### 5.1 动态规则生成

每次状态变化后重新编译 DNR session rules。

触发源：

- 插件启用 / 暂停切换。
- Env 配置变化。
- Tab 创建、关闭、移动。
- Tab Group 创建、更新、删除。
- active tab 变化。
- window focus 变化。

编译流程：

```text
load GlobalState
if global.enabled is false:
  remove all extension-owned session rules
  return

for each enabled Env:
  resolve linked Tab Groups
  collect tabIds
  compile filters to url conditions
  compile enabled headers to modifyHeaders actions
  create DNR session rules scoped by tabIds

updateSessionRules(remove old ids, add new ids)
```

### 5.2 Rule ID 分配

DNR 动态规则需要稳定数字 ID。建议维护一个插件内部 ID 区间。

```text
HEADER_RULE_BASE = 10000
每次全量重建时，从 HEADER_RULE_BASE 开始递增
ruleMeta.activeRuleIds 保存当前已安装规则 ID
```

```ts
type RuleMeta = {
  activeRuleIds: number[];
  lastCompiledAt?: number;
};
```

### 5.3 Header Rule 示例

```ts
const rule = {
  id: 10001,
  priority: 1,
  action: {
    type: "modifyHeaders",
    requestHeaders: [
      {
        header: "x-env-branch",
        operation: "set",
        value: "feature/checkout-fix"
      }
    ]
  },
  condition: {
    tabIds: [123, 124],
    urlFilter: "||pre.example.com/commerce/",
    resourceTypes: ["main_frame", "sub_frame", "xmlhttprequest", "fetch"]
  }
};
```

实际实现时需要按 Chrome DNR 支持的 condition 字段选择 `urlFilter` 或 `regexFilter`。简单域名和路径优先使用 `urlFilter`，复杂通配再使用 `regexFilter`。

## 6. URL Filter 编译

输入：

```json
{
  "domains": ["pre.example.com", "*.example.org"],
  "paths": ["/commerce/*", "/api/*"],
  "excludedDomains": ["sso.example.org"]
}
```

输出策略：

- 每个 domain/path 组合生成一条 DNR rule。
- excluded domain 通过不生成规则或额外校验处理。
- 对于 `*.domain.com`，生成匹配子域的 filter。

示例：

```text
pre.example.com + /commerce/* -> ||pre.example.com/commerce/
pre.example.com + /api/*      -> ||pre.example.com/api/
```

对于复杂情况，可降级为 `regexFilter`：

```text
^https?://([^/]+\.)?example\.org/(commerce|api)/
```

## 7. Tab Group 绑定

### 7.1 绑定当前 Group

用户在 Popup 中点击 `Bind current group`：

```text
get active tab
if activeTab.groupId === -1:
  prompt user to create group or disable bind
else:
  get tab group by groupId
  create/update GroupBinding
  append groupKey to Env.linkedGroupKeys
  recompile DNR rules
```

### 7.2 多 Group 绑定同一 Env

允许一个 Env 绑定多个 Group。

不允许一个 Group 同时绑定多个 enabled Env。若用户尝试绑定：

```text
This tab group is already bound to Env "A".
Replace with Env "B"?
```

### 7.3 浏览器重启后的恢复

Chrome 重启后，原始 `tabGroupId` 可能变化。插件启动时做恢复：

```text
query all tabs
group by tab.groupId
read group title/color
match existing GroupBinding by title + known URL
update chromeGroupId/windowId
recompile DNR rules
```

恢复不应强行猜测。若无法唯一匹配，标记 binding 为 `unresolved`，在 UI 中提示用户重新绑定。

## 8. Popup UI 状态

### 8.1 自动切换

当 `autoSwitch = true`：

```text
on active tab changed:
  resolve active tab group
  find env bound to group
  set selectedEnvId
```

当 `autoSwitch = false`：

```text
active tab changed does not mutate selectedEnvId
```

规则注入不依赖 `selectedEnvId`，只依赖 GlobalState 中的 Env 和 GroupBinding。

### 8.2 当前页面命中状态

Popup 打开时计算：

```text
active tab URL
active tab group
bound Env
filter matched?
headers count?
global enabled?
env enabled?
```

展示状态：

- Active：当前页面会注入规则。
- Not matched：当前页面不在过滤器范围内。
- No Env：当前 Tab Group 未绑定 Env。
- Paused：插件已暂停。
- Env disabled：Env 已禁用。

## 9. Import / Share

### 9.1 导出格式

```json
{
  "version": 1,
  "type": "browser-group-env.env",
  "env": {
    "name": "Checkout Fix",
    "enabled": true,
    "filters": {
      "domains": ["pre.example.com"],
      "paths": ["/commerce/*"],
      "excludedDomains": []
    },
    "rules": {
      "headers": [
        {
          "enabled": true,
          "name": "x-env-branch",
          "value": "feature/checkout-fix"
        }
      ],
      "queries": [],
      "cookies": []
    }
  }
}
```

默认不导出 `linkedGroupKeys`，避免导入后错误绑定到别人的浏览器。

### 9.2 脱敏导出

脱敏策略：

```text
Header value -> "***"
Cookie value -> "***"
Query value  -> 可配置，默认保留
```

### 9.3 导入策略

导入时支持：

- Duplicate：创建新 Env。
- Replace：替换同名 Env。
- Merge：合并规则和过滤器，不修改绑定。

## 10. Query 与 Cookie 后续实现

### 10.1 Query

Query 改写不适合直接用 Headers 的同一条 DNR 路径处理。可选方案：

- 使用 `declarativeNetRequest` redirect / transform 能力。
- 限制只对 main_frame 和部分请求类型生效。
- 对复杂 URL 改写给出明确限制。

建议第二阶段再做。

### 10.2 Cookie

Cookie 写入可以使用 `chrome.cookies` API，但需要新增权限。

风险：

- Cookie 对登录态和鉴权影响更大。
- SameSite / Secure / Domain 行为容易被误解。
- 删除或暂停时是否回滚 Cookie 需要明确设计。

建议第三阶段再做，并默认不自动清理用户已有 Cookie。

## 11. Codex Skill 对接点

Skill 是低优先级辅助能力，不属于插件 MVP。

未来可以增加本地配置：

```json
{
  "taskId": "http-header-worktree",
  "envName": "Checkout Fix",
  "groupTitle": "Codex: checkout",
  "urls": ["https://pre.example.com/foo"],
  "headers": {
    "x-env-branch": "feature/checkout-fix"
  },
  "scope": ["https://pre.example.com/*"]
}
```

Skill 行为：

```text
读取当前 worktree 的 .codex/browser-context.json
调用本地 native helper 或 Chrome extension external messaging
创建或聚焦对应 Tab Group
可选导入/更新 Env 配置
```

不建议让 Skill 直接操作 Header 注入。Header 注入应始终由 Chrome 插件统一负责。

## 13. 关键测试用例

### 13.1 Header 注入

- 当前 Tab Group 绑定 Env 且 URL 命中 Domain，Header 被注入。
- 当前 Tab Group 绑定 Env 但 URL 不命中 Domain，Header 不注入。
- 当前 Tab 不在任何 Tab Group，Header 不注入。
- 插件 Paused 时，Header 不注入。
- Env disabled 时，Header 不注入。

### 13.2 Tab Group 行为

- 一个 Env 绑定多个 Tab Group，多个组均注入同一 Header。
- 一个 Tab Group 尝试绑定第二个 Env，触发替换确认。
- 删除 Env 后，对应 Tab Group 不再注入 Header。
- 移动 Tab 到另一个 Group 后，规则跟随新的 Group 重新计算。

### 13.3 过滤器

- `pre.example.com` 只匹配该域名。
- `*.example.com` 只匹配子域名。
- `Excluded Domains` 优先排除。
- Path 为空时匹配全部路径。
- Path 不为空时只匹配指定路径。

### 13.4 Import / Share

- 导出 JSON 可以重新导入。
- Redacted 导出不会包含敏感 Header value。
- Replace 不修改已有 Group 绑定，除非用户明确选择。

## 14. 已知限制

- Chrome DNR 动态规则存在数量限制，需要控制规则编译粒度。
- Tab Group 恢复依赖标题和 URL 推断，不保证 100% 自动恢复。
- 复杂 URL 改写和 Cookie 管理不进入 MVP。
- 插件无法直接感知 Codex App 当前选中的对话；Codex Skill 或 Hook 只能作为外部触发入口。
